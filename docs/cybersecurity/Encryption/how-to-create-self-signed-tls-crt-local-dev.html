How to Create Self-Signed TLS Certificate for Local Development

To enable secure communication (TLS/HTTPS) between a server (like an EC2 instance) and a client (e.g. your MacBook) in a development environment, we can create our own trusted certificates.

This guide walks through a three-phase process on a macOS machine:

  1. Create a Certificate Authority (CA): You'll act as your own trusted authority, like a mini-Let's Encrypt.

  2. Issue a Server Certificate: Your new CA will sign a certificate for your specific domain.

  3. Trust Your CA: You'll tell your Mac's operating system to trust your CA, making the certificates it issues valid in browsers like Chrome.

(Optional) Phase 1: Create Your Certificate Authority (CA)​

The CA is the root of trust. It's like creating your own local "DMV" that has the power to issue trusted ID cards (TLS certificates). If you already have a CA created, continue to Phase 2.

Step 1.1: Create the CA's Private Key​

This command generates a private key for your CA. This key is the foundation of your CA's identity and must be kept secret. It's the "master stamp" used to sign and validate certificates.

openssl genrsa \
-out ca.key.pem \
2048

Step 1.2: Create the CA's Root Certificate​

Using the private key, we now create a public "root certificate." This file (ca.pem) is the public face of your CA. You'll share this with clients (like your Mac) so they know to trust certificates signed by your CA's private key.

openssl req \
-x509 \
-new \
-nodes \
-key ca.key.pem \
-sha256 \
-days 1825 \
-out ca.pem \
-subj "/C=IL/ST=Jerusalem/L=Jerusalem/O=kbbgl.github.io/OU=Engineering/CN=kbbgl.github.io dev"
  • -x509: Specifies the output should be a self-signed certificate, not a request.
  • -new: Indicates this is a new certificate.
  • -nodes: "No DES," meaning don't encrypt the private key with a passphrase.
  • -key ca.key.pem: The private key to use for signing.
  • -days 1825: Sets the certificate's validity period (5 years).
  • -out ca.pem: The output file for your new public root certificate.
  • -subj: Provides the identity information for the certificate non-interactively.

Step 1.3 Install and Trust the CA​

Your Mac doesn't know about your new CA yet. This command adds your CA's public root certificate (ca.pem) to the system's list of trusted authorities. You will be prompted for your password.

Bash

sudo security add-trusted-cert \
-d \
-r trustRoot \
-k "/Library/Keychains/System.keychain" \
ca.pem

After this, any certificate signed by your CA will be automatically trusted by browsers and other applications on your Mac, resulting in a green padlock icon.

Phase 2: Issue a Certificate for Your Server​

Now that you have a CA, you can issue a certificate for your server (kbbgl.github.io).

First, set an environment variable for the domain you'll be securing. This makes the following commands easier to copy and paste.

export DOMAIN_NAME=kbbgl.github.io

Step 2.1: Create the Server's Private Key​

Your server needs its own unique private key, separate from the CA's.

openssl genrsa \
-out $DOMAIN_NAME.key.pem \
2048

Step 2.2: Create a Certificate Signing Request (CSR)​

The CSR is the formal "application" from your server to your CA asking for a signed certificate. It bundles the server's identity information (like its domain name) and its public key.

openssl req \
-new \
-key $DOMAIN_NAME.key.pem \
-out $DOMAIN_NAME.csr \
-subj "/C=IL/ST=Jerusalem/L=Jerusalem/O=kbbgl.github.io/OU=Engineering/CN=$DOMAIN_NAME"
  • CN=$DOMAIN_NAME: The Common Name must be your server's primary domain name.

Step 2.3: Create the Certificate Extension File​

This configuration file provides essential modern details for the certificate, most importantly the Subject Alternative Name (SAN). Browsers today require the SAN field to validate a domain. This is where you can list multiple domains, including wildcards.

cat <<EOF > $DOMAIN_NAME.ext
authorityKeyIdentifier=keyid,issuer
basicConstraints=CA:FALSE
keyUsage = digitalSignature, nonRepudiation, keyEncipherment, dataEncipherment
subjectAltName = @alt_names
[alt_names]
DNS.1 = $DOMAIN_NAME
DNS.2 = *.$DOMAIN_NAME
EOF

Step 2.4: Sign the Server Certificate with Your CA​

This is the final step. Your CA will now use its private key (ca.key.pem) to verify and sign the server's CSR, creating the official, public TLS certificate (.crt file). This is the file you will install on your web server (e.g., Nginx).

openssl x509 \
-req \
-in $DOMAIN_NAME.csr \
-CA ca.pem \
-CAkey ca.key.pem \
-CAcreateserial \
-out $DOMAIN_NAME.crt.pem \
-days 825 \
-sha256 \
-extfile $DOMAIN_NAME.ext
